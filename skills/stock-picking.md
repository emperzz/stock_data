# stock-picking

A 股选股 skill。**流程执行器**——用户有明确选股需求时触发，按选股漏斗从强势板块收敛到具体可买个股候选，输出 chat 精简候选表 + 写入 `stock_picks/` 目录文件。

> **核心约束（来自 market-principles）**：本 skill 不绑定任何特定数据 API。正文只描述**每一步需要哪类数据**（板块行情、涨停股池、个股 K 线、公司画像…），**具体端点、参数、source 一律去 `market-data-obtain` 查**——取数时以那里为准，不要凭记忆猜端点。所有判断标准（龙头识别、量价、风险第一）走 `market-principles`，本 skill 只在此之上补充**选股特有的门槛与步序**。

---

## 1. 适用场景

满足以下任一情况时启用本 skill：

- 用户明确要求选股 / 挑票 / "买什么票"
- 用户想要"跟随某龙头 / 某主线的可买跟随票"
- 用户指定某个板块，要求在板块内挑票

**不适用的请求**：

- 市场判断 / 复盘 / 归因 → 走 `market-recap` / `market-principles`
- 仓位 / 止损 / 加减仓 → 交易决策层，本 skill 不覆盖
- 单股深度基本面研究 → 非本 skill 范畴（本 skill 隐含短线视角）

**与 `market-recap` 不冲突**：两者不同触发、不同产物、不同目录，仅有 `recap → market_tracking.md → 本 skill 只读富化` 的单向数据流。本 skill 实时自采板块为主，**不要求 recap 先跑过**。

---

## 2. 入口双分支

agent 收到选股请求后先判断用户是否**指定了板块**：

| 分支 | 触发 | 行为 |
|---|---|---|
| **A. 未指定板块**（默认） | "帮我选票" / "龙头带动的票" | 走完整板块漏斗（§3 步骤 1-5）→ 板块内选股（§4） |
| **B. 指定板块** | "在 X 板块选股" / "X 板块里有什么票" | **跳过板块强势度检测**，直接定该板块当前龙头 → 板块内选股（§4） |

**分支 B 的龙头处理**：

1. 取指定板块涨停股，按 `market-principles §6` 定当前龙头
2. **有龙头** → 正常走 §4（含业务与龙头比对）
3. **无龙头**（板块内无涨停）→ 无对标锚点，**跳过业务比对（§4 步骤 9），纯按 K 线/技术面选股**（§4 步骤 6-8）

> 分支 A 里能过漏斗的板块必含涨停股 → 必有龙头，"无龙头"只出现在分支 B。

## 3. 板块阶段（分支 A / 默认）

**实时数据为主（权威闸门），tracking 只做事后富化（不增删板块集）。** 每步"要哪类数据"如下，**取数端点去 `market-data-obtain` 查**（板块类数据推荐用其建议的 source）。

### 步骤 1：板块排名筛涨幅
- 要的数据：**带行情的板块清单**（含涨幅），按涨幅排序
- **硬门槛**：涨幅 >1%；**临场**：取排名靠前的

### 步骤 2：取筛后板块的成分股清单
- 要的数据：通过板块的**成分股清单**

### 步骤 3：板块内涨停股 + 涨停结构门槛
- 要的数据：**涨停股池**，看板块内涨停分布
- **硬门槛**：涨停结构达 **3 个 10cm 或 1 个 20cm + 2 个 10cm**
- **临场**：板块股数"不太少"（太小的板块资金量不足，大资金不感兴趣）

### 步骤 4：重叠板块去重
- 对成分股**重叠率高**（临场，参考交集/并集 >50%）的板块，**只取涨幅靠前的那一个**
- 细分板块辨识：把涨停最多的板块里的票细细归到对应细分板块

### 步骤 5：过闸板块富化 + 定龙头 + 归因
- **富化**：去 `market_tracking.md` 匹配活跃主线，命中则贴主线上下文（`ml:<slug>`、核心结论、连板梯队），并标注"延续既有主线 vs 新主线"。**tracking 不增删板块集，只加注释。**
- **定龙头**：按 `market-principles §6` 在板块涨停股内做相对比较
- **归因**：搞清板块上涨 + 龙头涨停的原因（外部事件型原因**必须**走 `market-data-obtain` 的新闻/公告/资金流能力后总结，禁止凭先验编造——见 `market-principles §8`）

## 4. 个股阶段（板块内漏斗）

在每个强势板块内，按以下顺序逐票过滤（前置阶段淘汰的不进入后续）。每步"要哪类数据"如下，**取数端点去 `market-data-obtain` 查**。

### 步骤 6：技术面（K 线）
- 要的数据：**个股日 K**（约近 30 日，可叠加均线指标）
- 优先：短期 K 线**多头排列或走平**、**无顶部压力**、K 线**离 5 日均线不远**（临场）
- 最佳形态：K 线与均线走平甚至多头排列 + 一根**放量大阳线**

### 步骤 7：分时量价
- 要的数据：**个股 1 分钟 K 线**
- 关注**价格涨跌对应的成交量**——分时能看到放量拉升（真拉升 vs 缩量诱多）

### 步骤 8：量价换手（硬门槛）
- 要的数据：**个股实时行情**（含市值 / 换手率 / 成交额）
- **中票**：换手≈5% 且 成交额≥10 亿
- **小票**：换手 10-20% 左右
- **临场**：尽量不碰大市值票
- **例外**（回引 `market-principles §5.2`）：多日小阳线 + 均线多头排列（健康缓涨）可放宽量能/换手要求

### 步骤 9：业务相似（仅在有龙头时）
- 先用**细分板块**粗筛（同细分赛道）
- 再对粗筛候选取**公司画像（主营业务）**，把主营与**龙头的主营**精确比对
- 挑主营与龙头**最相似**的（受带动作用最强，回引 `market-principles §6.6`）
- 筛掉"同板块不同赛道"的伪相似（如半导体板块里设备 vs 材料 vs 设计）

### 步骤 10：财务（末位 tiebreaker）
- **仅当**前面筛完候选并列难取舍时才执行
- 对并列候选查盈利：先看**公司画像**里的盈利字段，缺失再 fallback 网络搜索（走 `market-data-obtain §3`）
- 优先选公司业务盈利的

## 5. 文件存储

### 5.1 目录结构
```
./stock_picks/
├── YYYY-MM-DD.md       # 当日选股（候选票 + 每票筛选依据/否决原因）
└── picks_tracking.md   # 可买候选 watchlist（跨日，仍有近期买点的票）
```

### 5.2 文件 vs Chat 分工

| 输出 | 内容 | 受众 |
|---|---|---|
| **chat 回复** | 精简候选表（代码/名称/所属板块+主线/入选理由一句话） | 当下沟通 |
| **`{date}.md`** | 完整版（每票走过哪几步、量价换手数、业务比对结论、否决票及否决原因、数据来源） | 事后验证 |
| **`picks_tracking.md`** | 仍有近期买点的候选（跨日累积；用 `ml:<slug>` 引用主线解耦） | 跨日追踪 |

### 5.3 `picks_tracking.md` 结构与失效规则

每条候选字段：`{代码, 名称, 所属主线 ml:<slug>, 入选日期, 入选逻辑, 当前买点状态}`。

**失效规则**（本 skill 独立于 `market_tracking.md` 龙头失效规则）——命中即**直接移除**：
- **技术破位**：跌破关键支撑 / K 线走坏
- **所属主线失效**：引用的 `ml:<slug>` 在 `market_tracking.md` 中已失效/移除
- **买点消失**：已大幅拉升脱离买入区间 / 逻辑兑现

> `picks_tracking.md` 与 `market_tracking.md` 的"持续关注的龙头股"语义不同：后者是市场记忆/主线锚点（失效绑 `market-principles §6.5` 龙头当天不涨停），前者是可买候选（多为跟随龙头的同业务二线票，未必涨停、未必是龙头）。二者靠 `ml:<slug>` 引用解耦，各管各的生命周期。

## 6. 工作流

1. 触发本 skill → 前置检查（交易日 / 时点，见 `market-principles §2`）
2. 判断入口分支（§2）：是否指定板块
3. 读 `market_tracking.md`（拿活跃主线做富化）；若今日 `market_recap/{date}.md` 存在可一并参考
4. 执行板块阶段（§3，分支 A）或直接定龙头（§2，分支 B）
5. 执行个股阶段漏斗（§4）
6. chat 出精简候选表 + 写 `stock_picks/{date}.md`
7. 更新 `picks_tracking.md`（新增仍有买点的候选 / 按失效规则清理旧候选）

## 7. Anti-patterns（不要做）

- **不要**凭记忆猜或写死端点 / 参数 / source —— 取数时去 `market-data-obtain` 查当前端点（服务器端点会变，硬编即漂移）
- **不要**重新定义龙头识别 / 量价原则 / 风险原则 —— 回引 `market-principles §6 / §5.2 / §5.1`
- **不要**把 tracking 主线当"强势板块判定结论" —— tracking 只富化，强弱一律用实时数据重算（分支 A）
- **不要**把硬门槛和临场判定混写 —— 数字明确的（涨幅 >1%、涨停结构、换手/成交额）严格执行，程度词（排名靠前、离 5 日线不远、尽量不碰大市值）临场判定
- **不要**凭模型先验编造涨停/上涨原因 —— 外部事件型原因走 `market-data-obtain` 后总结 + 标注来源
- **不要**用不同板块分类体系混选 —— 全程用同一套（`market-principles §6.2`）
- **不要**给出交易决策（"建议买入 X 股"/"止损位 N 元"/仓位）—— 本 skill 只产候选，买卖与仓位由用户决定
- **不要**把可买候选塞进 `market_tracking.md` 的龙头区 —— 语义/失效规则不同，用 `picks_tracking.md`
- **不要**为"最后考虑"的财务给每只票都拉数据 —— 财务只在候选并列时作 tiebreaker（§4 步骤 10）
- **不要**在分支 B（指定板块无龙头）时硬做业务比对 —— 无锚点则纯技术选股

## 8. 跨 skill 调用约定

收到选股请求时：

1. **先看本 skill frontmatter triggers** —— 匹配则激活
2. **数据采集跳读 `market-data-obtain`** —— 取端点 + fallback 策略
3. **判断标准跳读 `market-principles`** —— 取龙头识别（§6）、量价（§5.2）、风险第一（§5.1）
4. **富化跳读 `market_tracking.md`** —— 取活跃主线上下文（只读，不改）
5. **不要**把本 skill 当"什么都能答"的入口 —— 市场判断/复盘请求引导走 `market-recap` / `market-principles`
