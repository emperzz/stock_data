# market-principles

A 股市场判断的**总入口 skill**。本 skill 本身**不绑定任何特定数据 API**——agent 自行决定数据来源，但**所有市场数据获取都通过配套 skill `market-data-obtain`**（详见第 3 节"配套 Skills"和第 8 节"数据获取约束"）。

---

## 1. 适用场景

满足以下任一情况时启用本 skill：

- 用户请求对当日或近期市场进行了解、判断、总结
- 用户针对某只个股、某个板块请求看法
- 用户要求整理每日消息、新闻要点
- 用户在开盘前 / 盘中 / 收盘后请求判断
- 需要识别当前市场龙头、活跃主线、板块轮动

**不适用的请求**：

- 仓位管理、止损点、加减仓规则（属于交易决策层，本 skill 不覆盖）
- 美股、港股、外汇、期货、加密货币（市场范围超出 A 股）
- 长期价值投资、基本面研究（本 skill 隐含短线视角）

---

## 2. 市场范围与前置检查

### 2.1 交易时段

- **时区**：Asia/Shanghai
- **集合竞价**：09:15-09:25
- **连续竞价**：09:30-11:30 / 13:00-15:00
- **收盘**：15:00

### 2.2 交易日判断

通过 A 股交易日历判断当前是否为交易日。非交易日数据只能取到上一交易日。

### 2.3 K 线数据返回规则（硬约束）

| 当前时间 | 数据可取范围 |
|---|---|
| 交易时段内、**未开盘**（09:15 之前） | 上一交易日 |
| 交易时段内、**已开盘**（含集合竞价和连续竞价） | 当日最新 |
| **非交易时间**（15:00 之后、09:15 之前） | 上一交易日 |

### 2.4 外盘信息（可选输入）

美股收盘等隔夜信息可作为 A 股开盘判断的输入，但**不**展开美股交易时段细节。

## 3. 配套 Skills

本 skill 是**总入口**。具体数据获取、判断细化、选股等子任务由配套 skill 完成，避免在主 skill 中堆叠所有规则。

| Skill | 触发场景 | 职责 | 链接 |
|---|---|---|---|
| `market-data-obtain` | 任何市场数据获取请求（行情、资金、新闻、公告、研报、板块） | 列出全部服务器端点 + 服务器失败时的 fallback 策略 | [skills/market-data-obtain.md](./market-data-obtain.md) |
| `market-recap` | 复盘流程执行（盘前 / 盘中 / 盘后三时段统一触发；写入 `./market_recap/{date}.md` + `market_tracking.md`） | 复盘流程的工程化（读取顺序、写入策略、chat / 文件分工）；归因方法论仍由本 skill 提供 | [skills/market-recap.md](./market-recap.md) |
| `stock-picking` | （未来）基于市场判断的选股 | 在活跃主线中筛选标的 | — |

**调用约定**：

- 收到市场判断请求 → 先看本表 → 数据获取走 `market-data-obtain`，复盘执行走 `market-recap`
- 数据获取与复盘流程之外的工作（判断、watchlist 维护）由本 skill 负责
- `market-recap` 在执行归因时回引本 skill 的判断方法论，但不重复定义
- 新增配套 skill 时在本表追加一行

## 4. 文件管理

### 4.1 目录结构

- 单一目录，扁平结构
- 路径：`market_recap_dir`（agent 启动时由用户配置，默认 `./market_recap/`）
- 例：`./market_recap/2026-07-15.md` 和 `./market_recap/market_tracking.md` 同目录
- 与 `market-recap` skill **共用同一目录**——文件模板、覆写 / 追加协议对两个 skill 一致

### 4.2 两类文件

| 文件 | 命名 | 用途 |
|---|---|---|
| 每日 md | `YYYY-MM-DD.md`（按 A 股交易日） | 当日快照、消息、时点判断、复盘 |
| 持久化文档 | `market_tracking.md` | 跨日延续的"市场记忆"——活跃主线、龙头股、未兑现逻辑等 |

### 4.3 写入策略

| 文档块 | 策略 | 工具操作 |
|---|---|---|
| 每日 md 顶部"消息" / "复盘" 块 | **覆写** | Edit 替换该 section 内容 |
| 每日 md 时间戳"判断"块 | **追加** | Edit 在"判断演化日志"前插入新块 |
| 持久化文档 | **按需更新**（全量重写或 diff 增量均可） | 每次写每日 md 时视需要同步更新 |

### 4.4 写入前必做

每次写入前先 `Read` 文件，定位到合适的插入点，避免覆盖未处理内容。

### 4.5 任务前后协议

本 skill 所有触发场景（个股判断、板块判断、消息整理、复盘等）通用，与第 4.3 节的写入策略配合：

- **任务执行前**：先 `Read market_tracking.md`，了解活跃主线、龙头股、未兑现逻辑（详见第 10 节）
- **任务执行后**：根据需要更新 `market_tracking.md` 和当日 `YYYY-MM-DD.md`（写策略遵循第 4.3 节）

`market_tracking.md` 是跨日延续的"市场记忆"，本 skill 的所有判断都应建立在其最新状态之上。

## 5. 核心原则

六条原则构成判断的底层逻辑：

### 5.1 控制风险永远大于追求利润（最高原则）

本原则凌驾于 5.2-5.6 之上——任何具体规则的判断结果如果与本条冲突，以本条为准。

- **保守倾向**：判断不明朗时不要硬推荐，宁可错过
- **避免激进建议**：宁可错过机会，也不要因为"错误判断造成损失"或"判断正确但风险敞口过大导致承担更多损失"而给出建议

### 5.2 量价是一切的核心

- **短期大涨 + 量能不放大 = 警示信号**：短期内大幅上涨但成交量未放大甚至萎缩，需要警惕
  - **例外**：涨停时由于涨跌停制度限制买盘无法兑现，量能不放大是正常的，不是警示
- **缓涨 in uptrend = 健康信号**：均线多头排列 + 上升通道中，每日小阳线小幅推升 + 中等量能，是健康走势，不是警示
- 消息刺激的上涨**必须**配合 K 线图形（好图形 + 消息才有大涨）
- **反面**：消息发出来时往往就是利润兑现的时候（跌）

### 5.3 趋势优先于绝对价格

- 是否关注某只票的决策永远建立在对**未来**价格变化的判断上
- **反面**：纠结绝对价格（如"这只票太贵了"）是错的，市场都是走出来的

### 5.4 技术指标仅作参考

- MA、MACD、RSI、KDJ、WR、BIAS、CCI、ATR、OBV、ROC、DMI、SAR、KC 都是参考
- **不是**决策核心来源

### 5.5 龙头是相对比较的结果

龙头不是达成某指标后自动产生的标签，而是**板块内（或全市场）涨停股之间的相对比较**。详见第 6 节。

### 5.6 趋势分析的时框选择

- **长期趋势**（几周~几个月，< 1 年）：看日线，必要时看周线
- **短期趋势**（几日~几周）：看日线 + 分钟线（5 分钟、15 分钟等）

不同的时框对应不同的判断目的：长期用于判断主线/板块方向，短期用于择时入场。本 skill 重点是短期研究（见第 12 节边界），但中长期趋势是判断的输入。

## 6. 龙头股判断方法

### 6.1 硬前提

**当日涨停**是龙头判断的前提。不涨停的股票直接出局，不进入龙头候选。

### 6.2 板块分类一致性

- A 股板块分类有多套（同花顺 / 东方财富 / 申万等），**没有绝对对错**
- 关键规则：判断时**必须保持分类一致**——不能用 THS 分类的"这只票"和 EM 分类的"那只票"做对比
- agent 选定一套分类后，全部判断沿用同一套

### 6.3 判断流程

龙头判断本质上是**比较**，不是"达成某指标就自动是龙头"。

**步骤**：

1. **圈定候选池**：
   - 板块龙头：所选板块内所有当日涨停的股票
   - 市场龙头：全市场所有当日涨停的股票

2. **必要条件筛选**（从候选池里筛掉明显弱的）：
   - 连板数（累计）较高者优先
   - 当日涨幅达标（10cm 票 10%、20cm 票 20%）
   - 首次封板时间较早
   - 成交额较大
   - **排除项**：断板、破板、烂板 的不入选

3. **相对比较**：
   - 在候选池内按"连板数 > 涨幅 > 封板时间 > 成交额"的权重排序
   - **连板是突破性指标**：连板数高的票即使当日涨幅较低也可能优于连板数低的票
   - **涨幅要看涨停质量**：10cm 强封板（封单大、封板早、涨停稳定）通常优于 20cm 弱封板（封单小、封板晚）
   - **特殊阈值**：
     - 20cm 票 **2 连板 ≈ 板块绝对龙头**（20cm 票 2 连板很稀缺，一旦达成即最强）
     - 10cm 票需结合板块内对比——若板块内只有 1 只 2 连板，它就是龙头

4. **特殊处理 - 断板重启**：
   - 昨天断板（未涨停），今天再次涨停的票
   - 前提仍是今日涨停
   - 结合板块内对比：若板块内无 2 连板票，这只"断板重启"很可能是龙头

### 6.4 龙头类型

| 类型 | 对比范围 |
|---|---|
| **板块龙头** | 所选板块内所有当日涨停股 |
| **市场龙头** | 全市场所有当日涨停股 |

两者的判断方法相同，只是**对比范围不同**。

### 6.5 龙头失效标准

| 龙头类型 | 失效条件 |
|---|---|
| **板块龙头** | **当天没有涨停**（最严格） |
| **市场龙头** | 连续 2 日跌停或大幅下跌（10cm 票 7%+、20cm 票 15%+），**或** 龙头自身转向震荡行情 |

### 6.6 龙头对板块/市场的意义

- 龙头是市场/板块的标杆，是资金热点
- 其他资金"看着龙头买"
- 龙头不倒 → 板块还有行情
- **同板块内**：主营业务与龙头越相似的股票，受带动作用越强

### 6.7 术语

| 术语 | 定义 |
|---|---|
| **封板** | 涨停价被买盘封住，股价无法继续上涨 |
| **断板** | 原本连续涨停的股票，当天未涨停（连板中断） |
| **破板** | 涨停后被打开（封板被砸开） |
| **烂板** | 多次封板多次打开，涨停不稳 |

## 7. 交易约束（仅做判断相关）

> **本节明确排除**：仓位管理、止损点、加减仓规则——这些属于交易决策层，由用户自行决定。本节只覆盖**判断层面的硬约束**。

### 7.1 硬约束

1. **量价不配合的大涨不可信**——多日小阳线缓涨 + 中等量能（均线多头排列 uptrend）是好信号，不在本条约束范围内（详见第 5.2 节）
2. **消息已广泛传播时不要追**——消息发出来往往是短期利润兑现时点
3. **龙头失效即停止以其为板块判断依据**——按第 6.5 节的失效标准
4. **技术指标仅作参考**，不作核心决策来源

### 7.2 软方法论

- 决策永远基于对**未来**的判断，不在历史/现在的描述上
- 个股看所属板块，板块看龙头个股（组合判断）
- 不依赖单一信号——消息 + K 线 + 量 + 板块 + 龙头 综合判断

## 8. 数据获取约束

> **本节是本 skill 与配套 skill `market-data-obtain` 的接口**——具体端点列表、fallback 策略、典型场景的端点组合见 `market-data-obtain`。本节只规定**在市场判断场景下的硬约束**。

### 8.1 硬约束

1. **所有市场数据获取必须通过 `market-data-obtain` skill 描述的服务器端点**
   - 不绕过 skill 直接调用任何外部 API
   - agent 应优先通过 `/control/api-manifest` 反射端点列表，避免硬编码
   - 详细端点 → 见 [skills/market-data-obtain.md](./market-data-obtain.md)

2. **外部事件型原因（"为什么涨/跌"）强制走数据获取协议**
   - 必须先调用 `market-data-obtain` 的新闻 / 公告 / 资金流 / 龙虎榜等端点
   - 服务器失败时按 `market-data-obtain` 第 3 节 fallback 策略处理（网络搜索工具优先，网页抓取工具次之；具体工具名因 agent 平台而异）
   - fallback 后**总结**而非直接搬运，并标注来源
   - 区分**事实**（搜索结果中明确写出的）与**推断**（基于事实的二次推断）
   - 多源矛盾时**列出分歧**而非强行收敛

3. **禁止凭模型先验知识直接给出外部事件型原因**
   - 反例："根据我的判断，这只票涨是因为政策利好" → ❌
   - 正例："根据财联社早报，今日 X 板块受政策利好走强" → ✅（来源明确）

### 8.2 技术面 vs 事件面分工

| 维度 | 走哪条规则 | 原因 |
|---|---|---|
| **外部事件型原因**（政策 / 消息 / 资金 / 情绪 / 公告 / 研报） | 强制走 `market-data-obtain` 第 3 节 fallback 协议 | 这类信息时效性强、模型先验易过时、影响判断准确性 |
| **技术面原因**（量价 / K 线形态 / 技术指标） | 仍按第 5.2 节量价方法论判断 | 这是 skill 既定的判断框架，不需要 web search |

### 8.3 反模式

- ❌ "根据我的判断，这只票涨是因为..." → 凭模型先验，未走数据获取
- ✅ "根据财联社早报，今日 X 板块受政策利好走强..." → 走 `market-data-obtain` + 标注来源
- ❌ 服务器返回空数据 → 编造原因填补
- ✅ 服务器返回空数据 → 触发 fallback 协议（agent 自带的网络搜索 / 抓取工具）
- ❌ 所有 fallback 也失败 → 编造内容
- ✅ 所有 fallback 也失败 → 明确告知用户"未能获取到相关信息"，不强答

### 模板使用规则

- 顶部"消息"和"复盘"块用 Edit 工具的 replace 模式（每次覆写该 section）
- 时间戳"判断"块用 Edit 工具的 insert 模式（在"判断演化日志"前插入）
- "判断演化日志"按时间正序记录判断变化，14:00 块中如反转 9:30 判断，需在此处显式记录"HH:MM 修正 → ..."
- **首次**写入该日文件时，先 Write 完整模板骨架

### 写作要求

- **保留前因**：本次判断如果反转/强化了前次判断，**必须在研判块里复述前次判断**，让读者理解为何变化
- **文字描述区分**反转与强化（不靠 emoji 或特殊标记，靠自然语言）
- **"原因"块**：外部事件型原因需注明数据来源（哪个端点 / 哪次 fallback 的哪个搜索结果）

### 持久化文档的更新规则

- **更新时机**：每次写每日 md 时按需同步更新（不需要独立触发，不需要周末对账）
- **更新方式**：全量重写或 diff 增量均可，效果一样
- **淘汰触发**：
  - **活跃主线**：板块连续弱势 / 资金净流出 / 政策兑现 → 移到"已失效/已兑现"
  - **持续关注的龙头股**：板块龙头当天未涨停 / 市场龙头触发第 6.5 节的失效标准 → 移到"已失效/已兑现"
  - **未兑现逻辑**：事件兑现 / 超时 / 逻辑被证伪 → 移到"已失效/已兑现"
- "已失效/已兑现"中的条目保留一段时间后由用户自行决定是否删除（skill 不规定）

## 11. 工作流示例

**场景**：用户说"今日市场怎么看"

**agent 流程**：

1. 触发本 skill（`market-principles`）
2. 前置检查（第 2 节）：确认当前时间点、是否为交易日
3. 读取 `market_tracking.md`（持久化文档）——了解活跃主线、龙头股、未兑现逻辑
4. 读取 `YYYY-MM-DD.md`（如存在）——了解今日已记录内容
5. **通过配套 skill `market-data-obtain` 收集数据**（消息、行情、板块、资金）——遵循第 8 节数据获取约束
6. 按第 6 节方法识别今日龙头、活跃主线
7. 在 `YYYY-MM-DD.md` 中按时间戳追加当前判断（"原因"块注明数据来源）
8. 如有变化，同步更新 `market_tracking.md`
9. 输出判断结论（含前因、判断、原因——原因块走第 8 节数据获取协议）
